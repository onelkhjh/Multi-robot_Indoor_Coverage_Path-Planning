# SCoPP 논문형 맵 감독 검토 보고서

## 판정

**조건부 승인** — 구현 착수용 설계 초안으로는 사용할 수 있으나, 아래 차단 항목을 해소하기 전에는 첫 마일스톤 완료 또는 “SCoPP 논문 그대로 구현”으로 승인할 수 없다.

검토 범위는 `AGENTS.md`, `docs/paper_map_spec.md`, `docs/map_implementation_plan.md`, `docs/map_test_plan.md`이다. 구현 코드는 검토하거나 수정하지 않았다.

## 차단 항목 (Critical)

### C-1. 논문 근거가 미해결인 핵심 동작이 구현 기본값으로 고정됨

- 근거: `paper_map_spec.md`는 셀 포함 기준, grid 원점/축, perimeter 간격, 좌표 변환, FoV 성분을 미해결 항목으로 둔다.
- 설계: `map_implementation_plan.md`는 `fully_contained`, bbox 최소점 anchor, `W/8`, 단일 UTM, 단일 정사각 FoV를 기본 동작으로 고정한다.
- 해결(사용자 결정): 기본값은 `any_overlap`이다. AOI와 양의 면적으로 겹치는 cell을 포함하고 no-fly zone과 양의 면적으로 겹치는 cell은 제외한다. 경계 cell은 실제 AOI 교집합을 `coverage_geometry`로 보존한다. 이는 논문 사실이 아니라 연구 정책으로 보고한다.
- 조건: 공식 저장소/논문 알고리즘을 확인해 각 선택에 근거를 붙이거나, 실행 결과와 API에서 `paper_verified=false` 및 가정 목록을 강제 기록한다. 논문 프로필과 보수적 프로젝트 프로필을 분리한다.

### C-2. 빈 유효 셀 처리 계약이 정면 충돌함

- 설계: 구현 계획 §6은 빈 셀 집합을 `NoCoverageCellsError`로 실패시킨다.
- 테스트: 테스트 계획 §9는 유효 셀이 0개인 정상 geometry가 “명확한 빈 결과”를 내야 한다고 요구한다.
- 위험: 동일 입력에 성공과 실패를 동시에 요구하므로 구현 및 독립 검증의 합격 판정이 불가능하다.
- 조건: 공개 API 계약 하나를 선택하고 명세·계획·테스트를 일괄 수정한다. 권고는 저수준 `discretize_map`은 빈 결과를 반환하고, 실행 파이프라인/CLI가 필요 시 명시적 오류로 승격하는 방식이다.

### C-3. 복수 component 입력 모델이 스키마와 불일치함

- 명세: AOI는 하나의 simple polygon이며 분리된 복수 AOI component는 논문에 명시되지 않았다고 적는다.
- 테스트: `disconnected`와 `paper_like` fixture는 복수 AOI component를 직접 요구한다.
- 설계: `MultiPolygon effective area`는 정상 처리한다고 하지만, 입력 AOI가 MultiPolygon인지 no-fly subtraction 결과만 의미하는지 불명확하다.
- 위험: fixture를 스키마로 표현할 수 없거나, 논문에 없는 복수 AOI 지원이 필수 합격 기준으로 승격된다.
- 조건: `MultiPolygon`의 생성 경로를 명시한다. v1 입력이 단일 AOI라면 fixture는 no-fly subtraction으로 분리된 effective geometry로 만들고, 직접 복수 AOI 입력은 확장 기능으로 분리한다.

## 주요 항목 (High)

### H-1. 요구사항-설계-테스트 추적 행렬이 없음

`MAP-01`~`MAP-11`은 논문 근거만 있고 구현 모듈/API 및 테스트 ID와 연결되지 않는다. 반대로 이미지 회귀, property test, 예외 정책은 어떤 MAP 요구사항을 검증하는지 알 수 없다. 각 MAP ID에 설계 위치, 테스트 ID, 상태, 가정 ID를 연결해야 한다.

### H-2. 경계 정책별 불변조건이 서로 충돌함

테스트 공통 조건은 “AOI 밖을 점유하는 cell은 유효할 수 없다”고 하고, 속성 테스트는 모든 perimeter 표본이 비행 가능 영역에 있어야 한다고 한다. 이는 `fully_contained`에는 맞지만 `center_inside`와 `min_coverage`의 정의에는 맞지 않는다. 정책 공통 불변조건과 정책별 불변조건을 분리해야 한다.

### H-3. 좌표 변환이 논문 식과 다른 후보인데 검증 대상이 UTM으로 고정됨

명세는 논문의 Haversine/행렬 변환을 사실로 기록하면서 원점과 분해 절차를 미해결로 둔다. 구현 계획과 테스트는 단일 UTM을 사실상 기준 구현으로 삼는다. UTM 정확도 테스트는 공학적 타당성만 입증하며 논문 재현성을 입증하지 않는다. `paper transform`과 `engineering projection`을 별도 프로필로 검증하거나 공식 코드 근거를 확보해야 한다.

### H-4. no-fly 경계 접촉 안전 정책이 충분히 명시되지 않음

구현 계획은 선/점 접촉을 허용하고 테스트는 경계 정책 하나를 문서화하라고만 한다. 비행 안전과 coverage 의미가 섞여 있다. 접촉 허용은 논문 가정인지 프로젝트 가정인지 metadata에 남기고, strict/area-overlap 정책별 기대 집합을 둬야 한다.

## 보통 항목 (Medium)

### M-1. 과도하거나 취약한 합격 기준

- 무작위 polygon 500개 전부 통과는 생성기 유효성, 시간 예산, 축소 전략이 없어 품질 기준보다 임의 숫자에 가깝다.
- 동일/새 프로세스 각 10회 SHA-256 비교는 결정성 확인에는 유용하지만 직렬화 형식과 라이브러리 버전 고정 없이는 거짓 실패가 난다.
- 시각 IoU `0.995`, 논문 근사 IoU `0.90`, 시작점 2%는 annotation 오차와 산출 근거가 없다.
- 모든 테스트 100%와 실패/제외 테스트 0개는 환경 비지원 테스트의 명시적 skip까지 차단할 수 있다.

기준별 산출 근거, 실행 시간 예산, 버전 범위, 허용 skip 정책을 정의하고 구조 검증을 이미지 수치 검증보다 우선해야 한다.

### M-2. tolerance 체계가 분산됨

테스트 공통 절대오차 `1e-9`, FoV 상대오차 `1e-12`, 구현 면적 비교 `1e-12`, 좌표 왕복 `0.01 m`가 각각 존재하지만 연산별 적용 규칙이 없다. 단위와 스케일에 따른 tolerance 표를 하나의 문서/API 상수로 통합해야 한다.

### M-3. FoV 입력 계약이 다름

구현 API는 `fov_deg`만 받지만 테스트는 degree/radian 입력을 각각 요구한다. radian을 공개 입력으로 지원할지, 테스트에서 독립 기대값 계산에만 사용할지 결정해야 한다.

### M-4. hole과 no-fly zone의 모델 관계가 불명확함

계획은 시각 의미만 구분하면서 geometry에서는 모두 제외한다. 스키마 검증은 “AOI 밖 hole”, “겹치는 hole”을 요구하지만 입력 모델에 hole 필드가 명확히 드러나지 않는다. AOI interior ring과 별도 no-fly polygon의 스키마, 겹침, 우선순위를 명시해야 한다.

## 낮은 항목 (Low)

- `docs/paper_spec.md`를 향후 추적 문서로 언급하지만 현재 실제 파일은 `paper_map_spec.md`다. 이름을 통일해야 한다.
- `MAP-10`의 “경계 정점 표시”는 구현 계획의 기본 렌더 스타일/API에서 명시적 옵션이 없다.
- 논문 실험 면적 `MAP-11`은 지원 가능성만 있고 fixture 면적 검증 테스트가 없다.
- 테스트 계획의 marker 명령은 marker 등록과 기본 전체 실행 명령을 함께 명시해야 한다.

## 승인 전 필수 조치

1. C-1~C-3의 계약 충돌을 해소한다.
2. `MAP-01`~`MAP-11`에 설계/API 및 테스트 ID를 연결한 추적 행렬을 추가한다.
3. 논문 검증 완료 전 모든 미확정 동작을 가정 ID로 표시하고 결과 metadata에 기록한다.
4. 정책 공통 테스트와 `fully_contained`/`center_inside`/`min_coverage`별 테스트를 분리한다.
5. 좌표 변환과 FoV 정책에 대해 논문 프로필과 프로젝트 확장 프로필의 경계를 확정한다.
6. 수치·속성·이미지 합격 기준의 근거와 CI 실행 예산을 기록한다.

위 조건을 문서에 반영하면 구현 착수 승인을 유지할 수 있다. 논문 동일성에 대한 최종 승인은 공식 코드/원문 대조, 구현 결과, 독립 테스트 결과가 모두 연결된 뒤 별도로 판단한다.
